Why Git for Teams?

Understanding distributed version control

ü§ù The Team Collaboration Challenge

When multiple developers work on the same codebase, chaos can ensue without proper tools:

  • Overwriting work - Developer A saves over Developer B's changes
  • Lost history - No way to see what changed or revert mistakes
  • Integration nightmares - Combining everyone's work becomes a guessing game

Git Solves These Problems

Git is a distributed version control system that tracks every change to your code:

  • ‚úÖ Every developer has a complete copy of the project history
  • ‚úÖ Changes are tracked in commits with who, what, and when
  • ‚úÖ Branches let developers work independently without conflict
  • ‚úÖ Merging combines work intelligently

üîë Key Concepts

Term Definition
Repository (Repo) Your project folder with its complete history
Commit A snapshot of your code at a point in time
Branch An independent line of development
Remote A shared copy of the repo (usually GitHub)

Repository Basics

Setting up and cloning repositories

üì¶ Getting Started with a Repo

Option 1: Clone an Existing Repository

When joining a project, clone the remote repository:

# Clone a repository from GitHub git clone https://github.com/org/project-name.git # Navigate into the project cd project-name

Option 2: Initialize a New Repository

Starting a brand new project:

# Create project folder and initialize Git mkdir my-project cd my-project git init # Connect to a remote repository git remote add origin https://github.com/org/my-project.git

üìÅ Understanding Git's Structure

After cloning, your project has a hidden .git folder containing all version history:

my-project/ ‚îú‚îÄ‚îÄ .git/ # Git's database (don't edit!) ‚îú‚îÄ‚îÄ src/ # Your source code ‚îú‚îÄ‚îÄ docs/ # Documentation ‚îú‚îÄ‚îÄ .gitignore # Files Git should ignore ‚îî‚îÄ‚îÄ README.md # Project description
‚ö†Ô∏è Never Delete .git The .git folder contains your entire project history. Deleting it means losing all version control data.

üîß Essential Configuration

Set your identity so commits show who made them:

# Set your name and email (use your work email) git config --global user.name "Your Name" git config --global user.email "your.email@company.com" # Verify your settings git config --list

The Commit Workflow

Staging, committing, and writing good messages

üìù The Three Stages of Git

Stage Description Command
Working Directory Your actual files on disk (just edit files)
Staging Area Changes ready to commit git add
Repository Committed history git commit

üîÑ The Commit Workflow

# 1. Check what files have changed git status # 2. Stage specific files git add src/feature.js git add tests/feature.test.js # Or stage all changes git add . # 3. Commit with a descriptive message git commit -m "Add user authentication feature"

‚úçÔ∏è Writing Good Commit Messages

Commit messages are crucial for team collaboration. They explain why changes were made.

‚úÖ Good Messages

"Add password validation to signup form" "Fix null pointer exception in user service" "Update API endpoints for v2 compatibility" "Remove deprecated payment gateway code"

‚ùå Bad Messages

"Fixed stuff" "WIP" "asdfasdf" "changes"
üí° Commit Message Format Use the imperative mood: "Add feature" not "Added feature" or "Adding feature". Think of it as completing the sentence: "This commit will..."
Step 4 of 6

Branching Basics

Working on features without affecting others

üåø Why Branches Matter

Branches let multiple developers work simultaneously without stepping on each other's toes:

  • main/master - The stable, production-ready code
  • Feature branches - Where new development happens
  • Each developer works on their own branch
  • Changes merge back to main when ready

üîß Essential Branch Commands

# See all branches (* marks current branch) git branch # Create a new branch git branch feature/user-login # Switch to that branch git checkout feature/user-login # Create AND switch in one command (preferred) git checkout -b feature/user-login # Return to main branch git checkout main

üìã Branch Naming Best Practices

Prefix Use For Example
feature/ New functionality feature/user-dashboard
fix/ Bug fixes fix/login-error
hotfix/ Urgent production fixes hotfix/security-patch
refactor/ Code improvements refactor/api-cleanup
‚ö†Ô∏è Never Commit Directly to Main Always create a branch for your changes, then merge via pull request. This protects the main branch and enables code review.
Step 5 of 6

Syncing with Remote

Push, pull, and fetch explained

üîÑ Keeping in Sync

Your local repo and the remote (GitHub) need to stay synchronized:

Command Direction Purpose
git push Local ‚Üí Remote Upload your commits
git pull Remote ‚Üí Local Download & merge changes
git fetch Remote ‚Üí Local Download without merging

üì§ Pushing Your Changes

# Push current branch to remote git push origin feature/user-login # First push of a new branch (sets upstream) git push -u origin feature/user-login # After setting upstream, just use: git push

üì• Pulling Changes

# Get latest changes from remote git pull origin main # Pull changes into your feature branch git checkout feature/user-login git pull origin main
üí° Pull Early, Pull Often The longer you wait to pull, the more likely you'll have conflicts. Pull from main at least once a day, or before starting new work.

üìã Daily Workflow Summary

# Start of day: Get latest changes git checkout main git pull # Create or switch to your feature branch git checkout -b feature/my-task # Work, commit frequently git add . git commit -m "Add input validation" # Push your work git push -u origin feature/my-task # End of day: Pull latest main into your branch git pull origin main
