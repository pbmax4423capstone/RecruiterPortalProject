Why PRs Matter

The cornerstone of team collaboration

üîÑ What is a Pull Request?

A Pull Request (PR) is a formal request to merge your changes into another branch. It's where:

  • Code review happens - teammates examine your changes
  • Discussion occurs - questions, suggestions, alternatives
  • Tests run - automated checks validate your code
  • Knowledge transfers - team learns about new changes

üéØ Benefits of Pull Requests

Benefit How It Helps
Quality Gate Catches bugs before they reach main branch
Knowledge Sharing Team stays aware of changes across codebase
Documentation Creates a record of what changed and why
Mentorship Junior devs learn from review feedback
Accountability Clear ownership of changes
üí° PR Golden Rule Never merge your own PR without review (except hotfixes). Even senior developers benefit from a second pair of eyes.
Step 2 of 6

Creating a Great PR

Set your reviewers up for success

üìù Anatomy of a Good PR

1. Clear, Descriptive Title

‚ùå Bad ‚úÖ Good
"Fixed bug" "Fix: Prevent duplicate submissions on login form"
"Updates" "Feature: Add export to CSV functionality"
"WIP" "[WIP] Refactor user authentication - feedback needed"

2. Meaningful Description

Your description should answer:

  • What - What does this PR do?
  • Why - Why is this change needed?
  • How - How does it work? (for complex changes)
  • Testing - How was this tested?

3. Right Size

  • Ideal: 100-400 lines changed
  • Maximum: 500 lines (beyond this, split into multiple PRs)
  • One logical change per PR
‚ö†Ô∏è Large PRs Get Ignored Studies show reviewers spend less total time on 1000+ line PRs than on 200-line PRs. Keep them small for better reviews.

Step 4 of 6

Code Review Etiquette

Review with respect and purpose

üë§ As a Reviewer

Do:

  • Be kind - You're reviewing code, not the person
  • Explain why - Don't just say "change this," explain why
  • Ask questions - "What's the reasoning here?" vs "This is wrong"
  • Suggest, don't demand - "Consider using..." vs "You must use..."
  • Highlight positives - Call out good code too!

Don't:

  • Use harsh language or sarcasm
  • Nitpick on style (automate with linters instead)
  • Request changes on unrelated code
  • Leave dozens of tiny comments
üí° Helpful Comment Examples
  • "Nit: minor suggestion, not blocking" ‚Üí low priority
  • "Consider: alternative approach..." ‚Üí optional suggestion
  • "Question: Could this cause X?" ‚Üí request for clarification

üë§ As the Author

Do:

  • Assume good intent - Reviewers want to help
  • Explain your thinking - If you disagree, explain why
  • Say thank you - Appreciate thorough reviews
  • Learn from feedback - Every review is a learning opportunity

Don't:

  • Take feedback personally
  • Argue over minor style preferences
  • Ignore comments without responding

Merge Strategies

How to integrate your changes

üîÄ Three Ways to Merge

1. Merge Commit

git merge feature-branch # Creates a merge commit preserving all history
  • ‚úÖ Preserves complete history
  • ‚úÖ Clear where branches joined
  • ‚ùå History can look cluttered

2. Squash and Merge

git merge --squash feature-branch # Combines all commits into one
  • ‚úÖ Clean, linear history
  • ‚úÖ One commit per feature
  • ‚ùå Loses granular commit history

3. Rebase and Merge

git rebase main git checkout main git merge feature-branch # Replays commits on top of main
  • ‚úÖ Linear history with all commits
  • ‚úÖ No merge commits
  • ‚ùå Can be confusing
  • ‚ùå Rewrites history (dangerous if shared)

üìä Which Strategy to Use?

Scenario Recommended
Simple feature, few commits Squash and Merge
Important granular history Merge Commit
Need clean linear history Rebase and Merge
Not sure / team default Squash and Merge
‚úÖ Team Recommendation We suggest Squash and Merge as the default. It keeps main history clean while preserving the PR as the source of detailed history.

Pull Request Mastery!

You now know how to create effective PRs and conduct meaningful code reviews.

üìã Key Takeaways

  • Write descriptive PRs - Clear title, meaningful description
  • Keep PRs small - Under 500 lines when possible
  • Use templates - Standardize what information is included
  • Review with kindness - Critique code, not people
  • Squash for cleanliness - Keep main branch history readable

‚úÖ PR Checklist

  • Title clearly describes the change
  • Description explains what, why, and how to test
  • PR is appropriately sized (<500 lines)
  • Self-review completed before requesting review
  • All tests passing
  • At least one approval before merging
