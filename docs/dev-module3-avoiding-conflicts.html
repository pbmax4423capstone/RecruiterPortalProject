Why Conflicts Happen

Understanding the root cause

‚öîÔ∏è What Causes Merge Conflicts?

A merge conflict occurs when Git can't automatically combine changes. This happens when:

  • Same lines edited - Two people change the same line(s) differently
  • Adjacent edits - Changes too close together for Git to merge safely
  • File renamed/deleted - One person edits a file another person deleted
  • Long-running branches - Branches diverge significantly over time

üìä The Real Problem: Time

The #1 predictor of conflicts is how long branches stay separate:

Branch Age Conflict Risk Resolution Difficulty
Hours üü¢ Low Usually trivial
1-2 Days üü° Moderate Quick to resolve
3-7 Days üü† High May need discussion
1+ Week üî¥ Very High Can be painful
üí° Key Insight Most conflicts are preventable. The goal isn't to become great at resolving conflicts‚Äîit's to rarely have them in the first place.

Prevention Strategies

Best practices to avoid conflicts

üõ°Ô∏è The Big Four Prevention Tactics

1. Pull Early, Pull Often

# Start every coding session with: git checkout main git pull git checkout your-branch git merge main

Sync with main at least once daily, ideally more.

2. Keep Branches Short-Lived

  • Aim for branches that last hours to 2 days, not weeks
  • Break large features into smaller, mergeable pieces
  • Use feature flags to ship incomplete features safely

3. Make Small, Focused Commits

  • Each commit should do one thing
  • Easier to review, easier to resolve if conflicts occur
  • "Commit early, commit often"

4. Don't Reformat Entire Files

  • Reformatting creates conflicts with everyone else's changes
  • If formatting is needed, do it in a separate PR
  • Better: set up auto-formatting so everyone's code looks the same
‚úÖ Daily Sync Habit Before starting work each day: git pull origin main into your branch. This 10-second habit prevents hours of conflict resolution.

Communication Protocols

Coordinate with your team

üó£Ô∏è Talk Before You Code

Many conflicts happen because two people unknowingly work on the same thing:

  • Announce your work - Post in team chat what you're starting
  • Check who's working where - Look at open PRs before starting
  • Claim files/areas - If you're doing a large refactor, let others know
  • Review open branches - See what teammates are working on

üìã Sample Team Workflow

Before Starting Work

# Post in team Slack/Teams: "üöÄ Starting work on user authentication - will be touching: - src/auth/* - src/components/LoginForm.js - src/api/userService.js Anyone else working in these areas?"

During Work

  • Push your branch regularly (even WIP) so others can see it
  • If you see overlap, reach out immediately
  • Consider pairing or coordinating timing

When Conflicts Are Likely

  • Talk to the other developer before attempting to resolve
  • Understand the intent of both changes
  • Decide together how to combine them
‚ö†Ô∏è Don't Guess at Others' Intent If you're resolving a conflict with someone else's code and you're not 100% sure what they intended, ask them. A 2-minute conversation beats a broken feature.
Step 4 of 6

Recognizing Conflicts

What conflicts look like

üö® When Git Can't Merge

When you pull or merge and there's a conflict, Git will tell you:

$ git merge main Auto-merging src/components/Header.js CONFLICT (content): Merge conflict in src/components/Header.js Automatic merge failed; fix conflicts and then commit the result.

üìÑ Conflict Markers in Files

Git marks the conflicting sections with special markers:

function getGreeting(user) { <<<<<<< HEAD return `Hello, ${user.firstName}!`; ======= return `Welcome back, ${user.name}!`; >>>>>>> main }
Marker Meaning
<<<<<<< HEAD Start of YOUR changes (current branch)
======= Separator between versions
>>>>>>> main End of THEIR changes (incoming branch)

üîç Check Conflict Status

# See which files have conflicts git status # Output shows: Unmerged paths: (use "git add <file>..." to mark resolution) both modified: src/components/Header.js
Step 5 of 6

Resolving Conflicts

Step-by-step conflict resolution

üîß Resolution Process

Step 1: Understand Both Changes

Before editing, understand what each version is trying to do:

  • What was YOUR intent with your change?
  • What was THEIR intent with their change?
  • Can both intents coexist?

Step 2: Edit the File

Choose one of these approaches:

  • Keep yours - Delete their section and conflict markers
  • Keep theirs - Delete your section and conflict markers
  • Keep both - Combine the changes logically
  • Write new - Create entirely new code that satisfies both needs
# Before (conflicted) <<<<<<< HEAD return `Hello, ${user.firstName}!`; ======= return `Welcome back, ${user.name}!`; >>>>>>> main # After (resolved - combined both) return `Welcome back, ${user.firstName}!`;

Step 3: Mark as Resolved

# Stage the resolved file git add src/components/Header.js # Commit the merge git commit -m "Merge main, resolve Header.js conflict"

üõ†Ô∏è Using VS Code to Resolve

VS Code provides visual conflict resolution:

  • Accept Current Change - Keep your version
  • Accept Incoming Change - Keep their version
  • Accept Both Changes - Keep both (stacked)
  • Compare Changes - Side-by-side diff view
üí° Pro Tip After resolving, always test your code. A syntactically correct resolution can still be logically broken.

üöë Emergency: Abort the Merge

If things go wrong and you need to start over:

# Abort merge and return to pre-merge state git merge --abort # Or if you've already made changes you want to discard git reset --hard HEAD
‚ö†Ô∏è Warning git reset --hard will discard ALL uncommitted changes. Make sure that's what you want.
