Why Branch Strategy Matters

The foundation of team collaboration

üåø Without a Strategy: Chaos

When teams don't agree on branching conventions:

  • ‚ùå Developers step on each other's code
  • ‚ùå Main branch becomes unstable
  • ‚ùå No one knows which code is "production ready"
  • ‚ùå Merge conflicts multiply
  • ‚ùå Code reviews become afterthoughts

‚úÖ With a Strategy: Order

A defined branch strategy ensures:

  • ‚úÖ Main is always stable - deployable at any time
  • ‚úÖ Features are isolated - work doesn't interfere
  • ‚úÖ Changes are reviewed - PRs before merging
  • ‚úÖ History is clear - easy to track what shipped when
  • ‚úÖ Rollbacks are simple - can revert cleanly
üí° One Rule Above All main (or master) should always be deployable. If it's not, your strategy has failed.

Feature Branch Workflow

The most common team workflow

üìã How It Works

  1. Create a branch from main for each feature/fix
  2. Work on your branch - commit frequently
  3. Push your branch to remote
  4. Open a Pull Request to merge into main
  5. Get code review from teammates
  6. Merge after approval
  7. Delete the branch - it's done!

Branch Naming Conventions

Consistent names improve team communication

üìù Recommended Naming Pattern

<type>/<description> # Examples: feature/user-authentication fix/login-button-disabled hotfix/security-vulnerability refactor/api-cleanup

üìã Branch Type Prefixes

Prefix Purpose Example
feature/ New functionality or enhancement feature/dark-mode
fix/ Bug fixes (non-urgent) fix/date-format-issue
hotfix/ Urgent production fixes hotfix/payment-crash
refactor/ Code improvements (no behavior change) refactor/service-layer
docs/ Documentation only docs/api-readme
test/ Adding or updating tests test/user-service

üéØ Naming Rules

  • Use lowercase - feature/add-login not Feature/Add-Login
  • Use hyphens - fix/button-color not fix/button_color
  • Be descriptive but concise - 2-4 words is ideal
  • Include ticket number (optional) - feature/JIRA-123-user-search
  • Avoid special characters - only letters, numbers, hyphens, slashes
‚ö†Ô∏è Avoid These mybranch, test123, johns-stuff, WIP - these tell teammates nothing useful.
Step 4 of 6

GitFlow vs Trunk-Based

Choosing the right workflow for your team

üîÄ Two Popular Approaches

üå≥ GitFlow

Multiple long-running branches for different purposes.

  • main - production code
  • develop - integration branch
  • feature/* - new features
  • release/* - release prep
  • hotfix/* - urgent fixes

üéØ Trunk-Based

Short-lived branches, frequent merges to main.

  • main - the only long-lived branch
  • Feature branches last hours/days
  • Merge to main multiple times daily
  • Feature flags for incomplete work

üìä When to Use Each

Factor GitFlow Trunk-Based
Team Size Large teams Small to medium teams
Release Cadence Scheduled releases Continuous deployment
Code Review Extensive review cycles Quick reviews, small PRs
Complexity More complex Simpler
Best For Products with versions Web apps, SaaS
üí° Our Recommendation For most teams, simplified trunk-based is ideal: short-lived feature branches, quick PRs, frequent merges to main. Save GitFlow for products that need versioned releases.

Protecting Main Branch

Enforce quality with branch protection rules

üõ°Ô∏è Why Protect Main?

Branch protection rules prevent accidents and enforce standards:

  • No direct pushes - all changes go through PRs
  • Required reviews before merge
  • Status checks must pass (tests, linting)
  • Force pushes prohibited

‚öôÔ∏è Recommended Protection Rules

In GitHub: Settings ‚Üí Branches ‚Üí Add branch protection rule

Rule Setting Why
Require pull request ‚úÖ Enabled No direct commits to main
Required reviewers 1-2 Second pair of eyes
Dismiss stale reviews ‚úÖ Enabled New changes need re-review
Require status checks ‚úÖ Enabled Tests must pass
Require branch up-to-date ‚úÖ Enabled Must have latest main
Include administrators ‚úÖ Enabled Rules apply to everyone

# In GitHub, protect these branches: main ‚Üí Full protection (required reviews, status checks) develop ‚Üí Partial protection (required reviews) release/* ‚Üí Protected (no force push)
‚ö†Ô∏è Don't Skip Reviews Even for "small" changes. Today's "quick fix" is tomorrow's production bug.
