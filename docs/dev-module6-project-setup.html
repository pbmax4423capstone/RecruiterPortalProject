üìã Naming Conventions

Type Convention Example
Folders lowercase, hyphens user-profile
Files (general) lowercase, hyphens data-service.js
Components PascalCase UserProfile.js
Tests Match source + .test UserProfile.test.js
üí° Consistency is Key Pick a convention and stick to it. Inconsistent naming creates confusion and makes code harder to find.

Essential Files

Every repo needs these

üìÑ README.md

The front page of your project. Should include:

# Project Name Brief description of what this project does. ## Quick Start ```bash git clone [url] npm install npm start ``` ## Prerequisites - Node.js 18+ - npm 9+ ## Environment Setup Copy .env.example to .env and fill in values. ## Team - Lead: @username - Contributors: @user1, @user2

üö´ .gitignore

Files that should never be committed:

# Dependencies node_modules/ # Environment files (contain secrets!) .env .env.local # Build outputs dist/ build/ # IDE settings .vscode/settings.json .idea/ # OS files .DS_Store Thumbs.db # Logs *.log npm-debug.log*

üîê .env.example

Template for environment variables (no real values!):

# Database DATABASE_URL=postgresql://user:password@localhost:5432/dbname # API Keys (get from team lead) API_KEY=your_api_key_here SECRET_KEY=your_secret_here # Feature Flags ENABLE_NEW_FEATURE=false
‚ö†Ô∏è Never Commit Secrets API keys, passwords, and tokens should NEVER be in your repository. Use environment variables and keep .env in .gitignore.

Branch Protection

Guard your main branch

üõ°Ô∏è Why Protect Branches?

Branch protection rules prevent accidents and enforce quality:

  • Prevents direct pushes to main
  • Requires code review before merge
  • Ensures tests pass before merge
  • Maintains audit trail of changes

‚öôÔ∏è Recommended Settings

In GitHub: Settings ‚Üí Branches ‚Üí Add rule

Setting Recommended Why
Require pull request ‚úÖ Yes No direct pushes to main
Required approvals 1-2 Ensures code review
Dismiss stale reviews ‚úÖ Yes New changes need re-review
Require status checks ‚úÖ Yes Tests must pass
Require branches up-to-date ‚úÖ Yes Prevents merge conflicts
‚úÖ Minimum Viable Protection At a minimum, require at least one approval before merging to main. This single rule prevents many problems.
Step 4 of 6

CODEOWNERS

Automatic review assignment

üë• What is CODEOWNERS?

A file that automatically assigns reviewers based on which files are changed:

  • Changes to auth code? Auth team is auto-requested
  • Changes to UI? Frontend team is auto-requested
  • Ensures the right people review the right code

üìÑ Example CODEOWNERS File

Create at .github/CODEOWNERS:

# Default owners for everything * @team-leads # Frontend components /src/components/ @frontend-team # API and backend /src/api/ @backend-team /src/services/ @backend-team # Authentication (security-critical) /src/auth/ @security-team @backend-team # Documentation /docs/ @docs-team *.md @docs-team # Configuration files (careful changes!) package.json @team-leads .github/ @team-leads
üí° Pattern Matching
  • * matches all files
  • /path/ matches directory and contents
  • *.js matches all JS files
  • Later rules override earlier rules

Team Onboarding

Getting new members started

üìã New Developer Checklist

  • GitHub account added to organization
  • Added to appropriate teams
  • Repository access granted
  • Local environment set up (README followed)
  • Can run tests locally
  • VS Code extensions installed
  • Copilot access configured
  • First test PR submitted and merged
üìù CONTRIBUTING.md

Guide for how to contribute to the project:

# Contributing to Project Name ## Getting Started 1. Fork the repository 2. Create a feature branch: `git checkout -b feature/your-feature` 3. Make your changes 4. Run tests: `npm test` 5. Submit a pull request ## Code Style - Use ESLint configuration provided - Run `npm run lint` before committing - Follow existing patterns in codebase ## Pull Request Process 1. Update README if needed 2. Add tests for new features 3. Get at least one approval 4. Squash and merge ## Questions? Reach out to @team-lead or post in #dev-help
‚úÖ First PR Exercise Have new team members submit a small "hello world" PR (like adding their name to CONTRIBUTORS.md) to practice the workflow before working on real code.
